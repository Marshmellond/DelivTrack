# 外卖配送实时监控与运营管理系统 — 详细设计方案

---

## 一、项目概述

### 1.1 项目名称
**外卖配送实时监控与运营管理系统**
（O2O Food Delivery Real-time Monitoring & Operations Management System）

### 1.2 项目目标
构建一套完整的外卖配送实时数据分析与运营管理平台。通过 Python 模拟器生成外卖订单数据，经 Kafka 消息队列缓冲后由 Flink 实时计算引擎进行多维度聚合分析，结果写入 MySQL 数据库。FastAPI 提供数据查询接口与后台管理 CRUD 接口，前端基于 Next.js + ECharts 实现实时数据大屏与后台管理页面。

### 1.3 核心功能
- **数据大屏**：9 个实时可视化图表，500ms 刷新
- **后台管理**：用户/商家/骑手/菜品/订单的增删改查
- **用户认证**：JWT 登录注册，角色权限区分
- **数据模拟**：自动生成外卖订单数据（500 条/秒）

### 4.2 建表 SQL

#### 4.2.1 基础数据表（后台 CRUD）

```sql
-- 用户表（登录注册）
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password VARCHAR(255) NOT NULL,           -- bcrypt 加密
    phone VARCHAR(20),
    address VARCHAR(200),
    role ENUM('admin','user') DEFAULT 'user',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_username (username)
) ENGINE=InnoDB;

-- 商家表
CREATE TABLE merchants (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    category VARCHAR(50) NOT NULL,            -- 中餐/西餐/快餐/甜品/小吃
    address VARCHAR(200),
    phone VARCHAR(20),
    status ENUM('active','closed') DEFAULT 'active',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_category (category)
) ENGINE=InnoDB;

-- 骑手表
CREATE TABLE riders (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    phone VARCHAR(20),
    vehicle VARCHAR(20),                      -- 电动车/摩托车/自行车
    status ENUM('online','offline','delivering') DEFAULT 'offline',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_status (status)
) ENGINE=InnoDB;

-- 菜品表
CREATE TABLE menu_items (
    id INT AUTO_INCREMENT PRIMARY KEY,
    merchant_id INT NOT NULL,
    name VARCHAR(100) NOT NULL,
    price DECIMAL(8,2) NOT NULL,
    stock INT DEFAULT 0,
    category VARCHAR(50),
    status ENUM('on_sale','off_sale') DEFAULT 'on_sale',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (merchant_id) REFERENCES merchants(id),
    INDEX idx_merchant (merchant_id)
) ENGINE=InnoDB;
```

#### 4.2.2 订单流水表（Flink 写入 + 前端只读）

```sql
CREATE TABLE orders (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_no VARCHAR(32) NOT NULL UNIQUE,       -- UUID 订单号
    user_id INT NOT NULL,
    merchant_id INT NOT NULL,
    rider_id INT,                                -- 可为空（未分配时）
    items JSON,                                  -- [{"name":"宫保鸡丁","price":28,"qty":2}]
    total_amount DECIMAL(10,2) NOT NULL,
    delivery_fee DECIMAL(6,2) DEFAULT 5.00,
    distance DECIMAL(5,2),                       -- 配送距离(km)
    status ENUM('created','accepted','delivering','delivered','cancelled') DEFAULT 'created',
    city VARCHAR(50),
    district VARCHAR(50),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    accept_time DATETIME,
    delivery_time DATETIME,
    finish_time DATETIME,
    INDEX idx_status (status),
    INDEX idx_city (city),
    INDEX idx_merchant (merchant_id),
    INDEX idx_create_time (create_time)
) ENGINE=InnoDB;
```

#### 4.2.3 聚合结果表（Flink 5 秒窗口写入，前端大屏只读）

```sql
-- 全局摘要（单行覆盖更新，始终只有 1 行）
CREATE TABLE dashboard_summary (
    id INT PRIMARY KEY DEFAULT 1,
    gmv DECIMAL(14,2) DEFAULT 0,               -- 累计 GMV
    order_count BIGINT DEFAULT 0,                -- 累计订单数
    rider_online INT DEFAULT 0,                  -- 在线骑手数
    rider_delivering INT DEFAULT 0,              -- 配送中骑手数
    avg_delivery_time DECIMAL(6,2) DEFAULT 0,    -- 平均配送时长(分钟)
    cancel_rate DECIMAL(5,2) DEFAULT 0,          -- 取消率 %
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;

-- 区域统计（按城市+区域维度）
CREATE TABLE dashboard_region (
    id INT AUTO_INCREMENT PRIMARY KEY,
    city VARCHAR(50) NOT NULL,
    district VARCHAR(50),
    order_count BIGINT DEFAULT 0,
    gmv DECIMAL(14,2) DEFAULT 0,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_region (city, district)
) ENGINE=InnoDB;

-- 商家排行
CREATE TABLE dashboard_merchant_rank (
    id INT AUTO_INCREMENT PRIMARY KEY,
    merchant_name VARCHAR(100) NOT NULL,
    category VARCHAR(50),
    order_count BIGINT DEFAULT 0,
    gmv DECIMAL(14,2) DEFAULT 0,
    update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_merchant (merchant_name)
) ENGINE=InnoDB;

-- 时段趋势（保留最近 1 小时数据）
CREATE TABLE dashboard_hourly (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    time_slot DATETIME NOT NULL,                 -- 5 秒窗口的开始时间
    order_count INT DEFAULT 0,
    gmv DECIMAL(12,2) DEFAULT 0,
    avg_delivery_time DECIMAL(6,2) DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_time (time_slot)
) ENGINE=InnoDB;
```

## 五、Python 数据模拟器

### 5.1 目录

```

├── config.py          # Kafka / MySQL / Flink 连接配置
├── generator.py       # 外卖订单生成逻辑
└── run.py             # 入口，循环发送
```

### 5.2 生成逻辑

1. 启动时从 MySQL 读取现有数据：`SELECT * FROM merchants WHERE status='active'`、`SELECT * FROM riders WHERE status='online' OR status='delivering'`、`SELECT * FROM users`、`SELECT m.* FROM menu_items m JOIN merchants mer ON m.merchant_id=mer.id WHERE mer.status='active' AND m.status='on_sale'`
2. 每次生成订单时随机组合：1 个用户 + 1 个商家 + 1~3 款菜品 + 1 个骑手
3. 生成订单 JSON，发送到 Kafka Topic `delivery-orders`
4. 发送速率：每 0.05 秒发 25 条 → **500 条/秒**

## 六、Flink 实时作业

### 6.1 目录

```
flink/src/main/java/com/delivery/
├── Order.java            # 订单 POJO
└── DeliveryJob.java      # 主作业（5 条聚合管道）
```

### 6.2 5 条聚合管道

| # | 管道 | 聚合方式 | 输出表 | 写入方式 |
|---|---|---|---|---|
| 1 | 全局摘要 | windowAll 5s → GMV/订单数/骑手在线/均配送时长/取消率 | `dashboard_summary` | `REPLACE INTO` |
| 2 | 区域统计 | keyBy(city+district) → order_count/GMV | `dashboard_region` | `INSERT ... ON DUPLICATE KEY UPDATE` |
| 3 | 商家排行 | keyBy(merchant_name) → order_count/GMV | `dashboard_merchant_rank` | `INSERT ... ON DUPLICATE KEY UPDATE` |
| 4 | 时段趋势 | 复用摘要 → 追加 | `dashboard_hourly` | `INSERT` + `DELETE WHERE time_slot < NOW()-INTERVAL 1 HOUR` |
| 5 | 订单流水 | 逐条解析 → 写入 | `orders` | `INSERT` |

### 6.3 技术要点

- KafkaSource 消费 `delivery-orders`，Jackson 解析 JSON
- Processing Time 5 秒滚动窗口
- JDBC 连接 MySQL（HikariCP 连接池），不使用 Jedis（Redis 换 MySQL）
- `dashboard_summary` 始终只有 1 行（id=1），Flink 定期 REPLACE
- `dashboard_hourly` 写入后立即 DELETE 超过 1 小时的行，防止表无限增长

## 十一、启动顺序

```
1. 确保 VM 集群已启动（ZK → HDFS → Kafka → Flink）
2. MySQL 建库建表（执行建表SQL.sql）
3. 后台录入初始数据（至少 10 商家 + 50 菜品 + 50 骑手 + 1000 用户）
4. 打包提交 Flink 作业 jar
5. 启动 FastAPI：uv run uvicorn api.main:app --host 0.0.0.0 --port 8000
6. 启动 Next.js：pnpm dev
7. 启动模拟器：uv run python simulator/run.py
8. 打开 http://localhost:3000
```

## 十二、Flink 与 MySQL 交互说明

为何 Redis 换 MySQL 不会冲突：

| 对比 | Redis 方案 | MySQL 方案 |
|---|---|---|
| 写入方式 | HSET/ZINCRBY/LPUSH 原子操作 | REPLACE/INSERT ON DUPLICATE KEY UPDATE |
| 聚合表更新 | keyBy 后每条调用 Redis 命令 | keyBy 后每条执行 SQL |
| 连接管理 | JedisPool（单连接简单） | HikariCP 连接池（需要池化） |
| 前端读取 | redis-py → Python dict | SQLAlchemy → Python dict |
| 聚合表行数控制 | ZSET ZREMRANGEBYSCORE | DELETE WHERE time < NOW()-1h |

Flink 写入 MySQL 使用 `REPLACE INTO`（全局摘要）和 `INSERT ... ON DUPLICATE KEY UPDATE`（区域/商家/时段），保证 5 秒窗口触发时更新已有行而非插入重复行。
